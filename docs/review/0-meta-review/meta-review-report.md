## 1. 개별 리뷰 평가

### 1.1 Gemini Review

**잘 식별한 것:**
- 50x48x6 분류 체계가 "유기적 시스템에 관료적 행정 절차를 도입한 형태"라는 핵심 모순을 정확히 포착
- Drift 차단이 패러다임 시프트(Epiphany)까지 거부할 위험 — "Human Review 큐" 우회로 제안
- Recall 파이프라인의 동기적 학습 업데이트가 스케일 시 병목이 될 것이라는 엔지니어링 직관
- 시간 기반 가지치기의 롱테일 인사이트 손실 위험, 레이어별 반감기 차등 필요성

**놓친 것:**
- **측정/검증 체계에 대한 언급 전무.** "좋아지고 있는가?"를 물어보지 않았다
- BCM 메타포의 수학적 유효성 자체를 검토하지 않음 — 설계의 프레임워크를 수용한 채 내부에서만 비판
- 구현 순서의 문제 (측정이 마지막인 점)를 지적하지 않음
- 크로스-프로젝트 오염, 승격 게이트 사멸 등 운영 레벨 위험 누락

**한계/맹점:**
- 설계의 **전제를 수용**한 상태에서의 비판. "이 시스템이 애초에 이 정도의 복잡도가 필요한가?"라는 근본 질문이 없다
- 4개 이슈 모두 "개선 방향"을 제시하지만, 우선순위나 심각도 판정이 없다
- 뇌과학 메타포를 긍정적으로 수용 ("뇌과학 기전을 치환한 논리가 매우 탄탄") — 메타포가 설계를 왜곡할 가능성을 간과

### 1.2 GPT Review

**잘 식별한 것:**
- 가장 체계적인 구조: 강점 → 리스크 5개 → 해결책 4개 → 우선순위 P0-P2 → 성공/실패 기준
- R1(타입/관계 폭발)에서 **구체적 증상 예시** 제시 (`supports`가 `related_to`처럼 쓰이는 라벨 의미 붕괴)
- R2: BCM/UCB는 "수단"이며 "무엇을 최적화하는가"라는 목표 정의가 빠져있다는 지적
- R5: **크로스-프로젝트 오염** — 3개 리뷰 중 유일하게 식별. 패치 전환이 잡음 주입원이 될 위험
- 해결책 (A)~(D)가 모두 **구현 가능한 수준으로 구체적**: 코어+facet 2단 구조, 3축 레이어, 원본/파생 분리, exploit/explore 레인 분리
- 검증 3종 구분 (스키마/동작/품질)과 "품질 검증 없으면 복잡해지기만 하고 좋아진 느낌만 남는다"는 한 줄

**놓친 것:**
- BCM의 **수학적 적합성** 자체를 검토하지 않음 (Perplexity가 유일하게 지적)
- 구현 순서 문제를 지적하지 않음 — Phase 3 측정이 먼저여야 한다는 점
- "9개 AI 검증"의 방법론적 문제를 언급하지 않음
- 구체적 파라미터 (8.2회 임계값, 0.5 코사인, 허브 상위 10개 보호)에 대한 정량적 검토 없음

**한계/맹점:**
- Gemini와 동일하게 **설계 프레임워크를 수용한 내부 비판**. "이 복잡도가 정당화되는가?"보다 "어떻게 안정화할 것인가?"에 집중
- 해결책이 구체적이지만 **추가 복잡도를 도입하는 방향** (3축 레이어, exploit/explore 레인 분리 등). 단순화 방향의 제안이 부족
- "성공/실패 판정 기준"이 정성적 — "설명 재작성이 줄고 결정 재현이 늘어난다"는 측정 방법이 없으면 판정 불가

### 1.3 Perplexity/Sonnet 4.6 Review

**잘 식별한 것:**
- **가장 근본적인 비판.** 3개 리뷰 중 유일하게 설계의 전제 자체를 공격:
  - "메타포가 설계를 주도하고 있다" — 뇌과학이 사후 정당화로 작동
  - BCM의 **수학적 부적합**: 연속적 firing rate 모델을 이산적 recall 이벤트에 적용하는 것은 수학적으로 다른 것
  - θ_m 파라미터의 homeostatic/synaptic time scale 비율 문제 — 문서에 튜닝 계획 없음
- "9개 AI 검증"이 검증이 아님 — 같은 학습 데이터 기반 AI는 같은 blind spot 공유, 확증 편향 위험
- **구현 순서 역전 지적**: Phase 3(측정)이 Phase 0 이전이어야 한다
- 구체적 파라미터 도전: 8.2회 임계값의 출처 없음, 0.5 코사인은 너무 낮음, 허브 상위 10개 보호가 오류 고착화
- 핵심 질문: "3,230개 기억이 평범한 벡터 검색 + 간단한 가중치 업데이트로 이미 충분하지 않은가?"

**놓친 것:**
- **대안 제시가 약함.** GPT의 코어+facet 구조나 3축 레이어 같은 구체적 구조 개선안 부재
- 크로스-프로젝트 오염 위험 미언급
- 가지치기 규칙에 대한 세부 분석 없음 (Gemini가 더 상세)
- 운영 UX 관점 (모드/패치 전환의 실사용 위험) 미검토

**한계/맹점:**
- "단순한 버전으로 실험하라"는 방향은 옳지만, **이미 v2.0이 18개월 운영된 시스템**이라는 맥락을 충분히 반영하지 않음. 순수한 greenfield 설계 비판에 가깝다
- BCM 수학적 비판이 정확하지만, **실제 구현이 BCM을 얼마나 충실히 따르는지** vs **영감만 빌린 것인지** 구분하지 않음
- 전반적으로 "파괴적 회의주의"에 가까움 — 시스템의 기존 가치(3,255노드 그래프, enrichment 파이프라인)를 과소평가

### 2.1 합의 사항 (3개 리뷰 모두 동의)

| # | 합의 | 심각도 |
|---|------|--------|

| C2 | **코사인 유사도 0.5 drift 임계값이 문제적.** 오탐/미탐 동시 발생 위험 | 중간 |

### 2.2 분기 사항

| 주제 | Gemini | GPT | Perplexity |
|------|--------|-----|------------|
| **BCM 유효성** | 수용 (적용만 비판) | 목표 정의 부재 지적 | **수학적 부적합 주장** |
| **해결 방향** | 개선 (레이어별 차등) | 구조화 (코어+facet, 3축) | **단순화 우선** (15타입 실험) |

| **복잡도에 대한 태도** | 제어 가능하다고 봄 | 안정화 장치 추가 | 근본적 과잉 의심 |
| **구현 순서** | 언급 없음 | 언급 없음 | **역전 필요 주장** |

### 2.3 고유 인사이트

- **Gemini 고유:** Drift 차단 항목을 "Human Review 큐"로 우회하는 아이디어. Recall과 학습을 비동기 분리해야 한다는 아키텍처 제안. 가지치기에서 "흔적을 남기는" 소프트 아카이브 개념.
- **GPT 고유:** 크로스-프로젝트 오염 위험 (R5). Exploit/Explore 레인 분리. 3축 레이어 체계 (maturity/confidence/stability). 원본 불변 + 파생 가변 분리. "성공 = 설명 재작성 감소, 결정 재현 증가" 정의.
- **Perplexity 고유:** BCM의 연속 vs 이산 수학적 불일치. "9개 AI 합의 != 검증" 방법론 비판. Phase 3→Phase 0 순서 역전. "평범한 벡터 검색이 이미 충분한가?" 근본 질문. action_log가 최대 테이블이 될 파티셔닝 위험.

## 3. 아이디에이션 평가: 원래 설계 스펙은 건전했는가?

### 건전한 부분

1. **"기억 = 동역학"이라는 핵심 가정은 옳다.** 단순 저장소가 아닌 강화/약화/승격/가지치기가 있는 시스템이라는 설계 방향은 3개 리뷰 모두 인정했다 (GPT: "견고", Gemini: "탄탄").

2. **action_log를 "why trail"로 설계한 것은 장기 운영에 필수.** GPT가 직접 칭찬. 디버깅과 감사 가능성의 기반.

3. **enrichment 피드백 루프의 거짓 기억 위험을 정면 대응한 것은 선견지명.** GPT: "이 통제가 없으면 자기 환각 증폭기가 된다."

4. **모드 전환 (focus/auto/dmn) + 패치 시스템은 실사용 UX로 연결되는 좋은 설계.** recall 맥락을 사용자 의도에 맞추는 메커니즘.

### 과잉 설계인 부분

1. **50+48 온톨로지는 이론적 완결성을 추구하다 실용성을 잃었다.** 3개 리뷰 합의. LLM 분류기가 48개 관계를 안정적으로 구분하기 어렵고, 시간이 갈수록 라벨 의미가 붕괴된다. 15개 코어 + facet 확장이 더 현실적.

2. **BCM 메타포의 충실한 수학적 구현은 불필요했을 수 있다.** Perplexity의 지적이 가장 날카롭다 — 이산적 recall 이벤트에 연속적 firing rate 모델을 적용하는 것은 수학적으로 다른 것이다. BCM에서 **영감**을 받되 **수식**을 그대로 이식하는 것은 과잉.

3. **3-gate 승격 (SWR + Bayesian + MDL)은 단일 시스템 치고 과도한 통계적 장치.** GPT의 R4 경고대로 — 파라미터/실패 케이스 정의 없이는 "승격이 거의 안 일어남" 또는 "엉뚱한 승격" 중 하나로 수렴.

### 순서가 잘못된 부분

- Perplexity의 핵심 지적이 가장 중요하다: **"측정이 마지막"인 순서는 역순이다.** 현재 recall 품질 baseline 없이 복잡한 메커니즘을 쌓으면, 개선 여부를 판단할 수 없다. Phase 3의 NDCG 측정이 Phase 0 이전이었어야 한다.

### 종합 판정

> 설계 **의도**는 건전하다. 설계 **규모**가 과잉이다. 설계 **순서**가 역전되어 있다.
> 핵심 아이디어(동역학 기반 기억, enrichment 안전장치, 모드 전환)는 유지하되, 복잡도를 절반으로 줄이고 측정을 앞세웠어야 한다.

## 4. v2.1 구현 시사점

### 해결된 우려

| 우려 | 리뷰 출처 | v2.1에서의 해결 |
|------|-----------|----------------|
| 측정 체계 부재 | C3 (전체 합의) | Goldset 25쿼리 생성, NDCG baseline 측정 (@5=0.057) |
| action_log 미구현 | GPT (강점 인정) | action_log 테이블 구현, 모든 작업 기록 |
| BCM+UCB 구현 | Gemini/GPT | hybrid BCM+UCB 구현, recall에서 작동 |
| 승격 게이트 | GPT R4 | SPRT + 3-gate (SWR/Bayesian/MDL) 구현 |
| drift 탐지 | C2 (전체 합의) | drift threshold 0.5 구현, 차단 로직 작동 |
| 가지치기 | Gemini 4번 | Phase 6 pruning 구현 |
| hub 보호 | Perplexity | hub_monitor 구현 |

### 남아있는 우려 (미해결 또는 부분 해결)

| # | 우려 | 심각도 | 상태 |
|---|------|--------|------|
| U1 | **NDCG @5 = 0.057은 매우 낮다.** Perplexity의 "측정 먼저" 경고가 정확했음을 증명. 복잡한 메커니즘을 쌓았지만 recall 품질이 아직 증명되지 않음 | **Critical** | goldset 튜닝 예정 |
| U2 | **50+48 온톨로지 축소 미진행.** 3개 리뷰 합의 사항이지만 v2.1에서 그대로 유지. 분류 일관성 측정도 없음 | 높음 | 미착수 |
| U3 | **drift threshold 0.5 재검토 미진행.** Perplexity: "너무 낮음, enrichment 100개 실측 후 캘리브레이션 필요" | 중간 | 0.5 유지 |
| U4 | **BCM 파라미터 문서화 부재.** Perplexity: τ, 학습률, θ_m 초기값/업데이트 규칙을 수식으로 명시해야 | 중간 | 미착수 |
| U5 | **8.2회 승격 임계값의 근거 불명.** Perplexity: "시뮬레이션? 실험? 임의 설정?" | 중간 | SPRT로 대체했지만 내부 파라미터 출처 불명 |
| U6 | **크로스-프로젝트 오염 방지 미흡.** GPT R5. 패치 전환 시 프로젝트 경계 오염 | 중간 | access_control 구현했지만 오염 측정 없음 |
| U7 | **action_log 파티셔닝 전략 없음.** Perplexity: "6개월 후 DB 최대 테이블이 될 것" | 낮음 (현재) | 미착수 |
| U8 | **원본/파생 분리 미구현.** GPT (C) 제안. enrichment가 원본을 덮어쓰면 거짓 기억 공고화 | 높음 | 미착수 |
| U9 | **허브 보호 조건이 연결 수 기준.** Perplexity: 잘못 연결된 노드가 허브가 되면 영구 보호 | 중간 | hub_monitor 있지만 품질 기준 미포함 |

### 우선 행동 권고

**즉시 (P0):**
1. NDCG @5 = 0.057 원인 분석 — goldset 쿼리 자체가 문제인가, recall 파이프라인이 문제인가, 온톨로지 노이즈가 문제인가 분리 필요
2. 원본/파생 분리 설계 — enrichment rollback 가능성 확보

**단기 (P1):**
3. 온톨로지 축소 실험 — 현재 50타입을 15개 코어 + facet으로 매핑한 뒤 recall 품질 비교
4. drift threshold 캘리브레이션 — enrichment 샘플 100개 실측

**중기 (P2):**
5. BCM 파라미터 수식 문서화 + θ_m 안정성 범위 정의
6. action_log 시간 기반 파티셔닝 전략

## 5. 리뷰어들의 리뷰에 대한 메타 관찰

### 공통 한계
- 3개 리뷰 모두 **설계 문서만 보고 판단**했다. 실제 코드, 테스트, 운영 데이터를 보지 않았다. 이는 Perplexity 자신이 지적한 "9개 AI 검증 = 검증 아님"과 동일한 구조적 한계를 이 메타리뷰 자체도 공유한다.
- **개인 지식 그래프라는 특수 컨텍스트**를 충분히 반영하지 못함. 다수 사용자 시스템이 아니라 단일 사용자의 사고를 외부화하는 시스템이므로, "스케일 문제"의 임계점이 일반적 시스템보다 훨씬 높다.
- **기존 v2.0의 18개월 운영 데이터와 경험**을 모른 채 비판. "처음부터 다시 설계한다면" 관점이 지나치게 강하다.

### 리뷰 스펙트럼
- **Gemini:** 가장 온건. 설계를 수용하고 개선점 제안. 실무적이지만 근본 질문이 없다.
- **GPT:** 가장 구조적이고 실행 가능. 위험 식별 + 구체적 대안 + 우선순위. 그러나 복잡도를 줄이기보다 "더 나은 복잡도"를 제안하는 경향.
- **Perplexity:** 가장 근본적이고 도전적. 전제 자체를 공격. 그러나 파괴적 회의주의에 가까워 기존 시스템의 가치를 과소평가.

### 최적 활용법
세 리뷰를 **레이어별로 소비하는 것**이 가장 유효하다:
1. **방향성 검증:** Perplexity의 근본 질문들로 "이 복잡도가 정당한가?" 자문
2. **구조 설계:** GPT의 구체적 대안들 (코어+facet, 3축 레이어, 원본/파생 분리)을 구현 후보로
3. **엔지니어링 디테일:** Gemini의 비동기 분리, 소프트 아카이브 등 실무 개선안 적용

## 6. 최종 판정

| 항목 | 판정 |
|------|------|
| 설계 의도 | **건전.** 동역학 기반 기억, enrichment 안전장치, 모드 전환 — 핵심 아이디어 유효 |
| 설계 규모 | **과잉.** 50+48 온톨로지, 3-gate 승격, BCM 수식 이식 — 복잡도 대비 품질 향상 미증명 |
| 설계 순서 | **역전.** 측정이 마지막이었음. NDCG 0.057이 이를 증명 |
| v2.1 구현 | **기술적으로 완성도 높음** (117/117 테스트, 5781 ins). 그러나 핵심 질문 "실제로 더 좋아졌는가?"에 아직 답 못함 |
| 다음 단계 | **NDCG 개선이 모든 것에 선행.** 온톨로지 축소 실험, drift 캘리브레이션, 원본/파생 분리 순서 |

> **한 문장:** v2.1은 정교한 엔진을 조립했지만, 아직 시동을 걸어 도로에서 달려본 적이 없다 — 지금 필요한 것은 더 많은 부품이 아니라 주행 테스트다.
