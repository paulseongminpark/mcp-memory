# mcp-memory 온톨로지 가이드

> 이 문서는 mcp-memory의 온톨로지 시스템을 처음부터 끝까지 설명한다.
> 기술 용어는 최소화하고, 온톨로지를 처음 접하는 사람도 이해할 수 있도록 작성했다.
> 작성: 2026-03-05, 3라운드 아이디에이션 완료 시점

---

## 1. 문제: AI는 기억하지 못한다

Claude와 대화할 때, 매 세션은 백지에서 시작한다. 어제 무슨 결정을 했는지, 지난주에 어떤 실수를 했는지, 3개월 전에 세운 원칙이 뭔지 — AI는 모른다. 매번 처음부터 설명해야 한다.

인간의 뇌는 이렇게 작동하지 않는다. 당신이 "지난번에 그거" 라고만 해도 친구는 맥락을 안다. 그건 **기억이 연결되어 있기 때문**이다. 하나의 기억은 다른 기억과 엮여 있고, 맥락 속에서 의미를 갖는다.

mcp-memory는 이 문제를 해결한다. **AI에게 기억을 주는 것.** 그것도 그냥 텍스트 파일이 아니라, 뇌처럼 연결되고, 강화되고, 약화되고, 진화하는 기억.

---

## 2. 온톨로지란 무엇인가

"온톨로지"라는 단어가 어렵게 들리지만, 핵심은 단순하다.

**온톨로지 = "세상을 어떤 카테고리로 나눌 것인가"에 대한 약속.**

도서관을 생각해보자. 책을 아무 순서로 꽂으면 찾을 수 없다. 그래서 분류 체계를 만든다 — 문학, 과학, 역사, 철학. 이 분류 체계가 도서관의 "온톨로지"다.

mcp-memory에서 온톨로지는 **기억을 어떻게 분류하고, 어떻게 연결할 것인가**를 정의한다.

### 세 가지 구성 요소

**1. 노드 타입 (50개) — "이 기억은 뭔가?"**

| 이름 | 뜻 | 예시 |
|------|---|------|
| Decision | 결정한 것 | "RRF k=30으로 변경하기로 했다" |
| Failure | 실패한 것 | "enrichment가 환각을 만들어 임베딩이 오염됐다" |
| Pattern | 반복 확인된 규칙 | "checkpoint 전 다른 pane 확인 필요" |
| Principle | 원칙 | "STATE.md는 단일 진실 소스" |
| Insight | 깨달음 | "GPT vs Gemini의 근본적 설계 철학 긴장 발견" |
| Observation | 관찰한 것 | "Paul은 마찰 제거에 집착한다" |
| Value | 가치관 | "더, 더, 더 — 완성이 아닌 확장" |
| ... | (총 50개) | |

모든 기억은 이 50개 타입 중 하나로 분류된다. "이건 결정이야", "이건 실패 기록이야", "이건 원칙이야" — 이 구분이 있어야 나중에 "최근 실패 기록 보여줘"가 가능해진다.

**2. 관계 타입 (48개) — "이 기억들은 어떤 관계인가?"**

| 이름 | 뜻 | 예시 |
|------|---|------|
| supports | A가 B를 뒷받침함 | "BCM 직행 결정"은 "B-1 설계"를 supports |
| contradicts | A가 B와 모순됨 | "tanh 제안"은 "BCM 직행"과 contradicts |
| evolved_from | A가 B에서 발전함 | "v2.0 설계"는 "v1.0"에서 evolved_from |
| derived_from | A가 B에서 파생됨 | "SPRT 파라미터"는 "Wald 1945"에서 derived_from |
| ... | (총 48개) | |

기억은 혼자 존재하지 않는다. "이 결정은 저 원칙을 뒷받침한다", "이 실패는 저 패턴과 모순된다" — 이런 연결이 기억에 의미를 부여한다.

**3. 레이어 (6단계) — "이 기억은 얼마나 깊은가?"**

```
L5  Axiom      — 절대 변하지 않는 공리 (1개)
L4  Value      — 핵심 가치관 (5개)
L3  Philosophy — 철학, 세계관 (수십 개)
L2  Framework  — 프레임워크, 방법론 (수백 개)
L1  Pattern    — 반복된 패턴, 인사이트 (수백 개)
L0  Observation — 일회성 관찰, 신호 (수천 개)
```

**레이어는 피라미드다.** 바닥(L0)에는 "오늘 이런 걸 봤다" 같은 가벼운 관찰이 수천 개. 꼭대기(L5)에는 "이것만은 절대 바뀌지 않는다" 같은 공리가 1개.

가벼운 관찰(L0)이 반복되면 패턴(L1)으로 승격된다. 패턴이 여러 맥락에서 확인되면 프레임워크(L2)가 된다. 이 과정이 **아래에서 위로 올라가는 자연스러운 기억의 성숙**.

## 3. 현재 상태: 3,230개의 기억

mcp-memory에는 지금 이런 기억들이 쌓여 있다:

```
노드: 3,230개 (50가지 타입으로 분류)
엣지: 6,020개 (48가지 관계로 연결)
레이어: L0(관찰) ~ L5(공리)
프로젝트: orchestration, portfolio, tech-review, mcp-memory, monet-lab, daily-memo
```

이걸 시각화하면 **거미줄 같은 그래프**가 된다. 각 노드(기억)는 여러 다른 노드와 실로 연결되어 있고, 자주 함께 호출되는 기억 사이의 실은 두꺼워지고, 안 쓰는 기억 사이의 실은 가늘어진다.

## 4. 뇌과학이 왜 여기에 있는가

mcp-memory가 단순한 데이터베이스와 다른 점: **뇌의 기억 메커니즘을 모방한다.** 왜? 인간의 뇌가 수십만 년에 걸쳐 진화시킨 기억 시스템이 "가장 검증된 설계"이기 때문이다.

### 4.1 헤비안 학습 — "같이 발화하면, 같이 연결된다"

뇌에서 두 뉴런이 동시에 활성화되면 그 사이의 시냅스가 강해진다. 1949년 도널드 헵(Donald Hebb)이 제안한 원리.

**mcp-memory에서:** "recall에서 A와 B가 같이 검색되면, A-B 연결이 강화된다." 자주 함께 쓰이는 기억은 더 강하게 연결되고, 다음 검색에서 더 쉽게 같이 나온다.

문제는 **이것만으로는 불안정**하다는 것. 모든 연결이 계속 강해지기만 하면, 결국 모든 것이 모든 것과 연결된다 — 의미가 사라진다. 이걸 해결하는 게 다음 메커니즘.

### 4.2 BCM 규칙 — "적당히, 균형있게"

BCM(Bienenstock-Cooper-Munro) 규칙은 헤비안의 업그레이드 버전. 핵심 아이디어:

> 활성화가 임계값(θ)보다 높으면 → 연결 강화
> 활성화가 임계값보다 낮으면 → 연결 약화
> 그리고 임계값 자체가 전체 활성화 수준에 따라 움직인다.

이게 왜 중요한가: **자주 쓰는 기억만 살아남고, 안 쓰는 기억은 자연스럽게 약해진다.** 뇌가 수면 중에 하는 "시냅스 정리"와 같은 원리.

**mcp-memory에서:** 각 기억 노드마다 개인 임계값(θ_m)이 있다. 자주 recall되면 θ_m이 올라가서 다음 강화가 더 어려워진다 (과잉 강화 방지). 안 쓰이면 θ_m이 내려가서 약한 활성화에도 연결이 약화된다 (자연 망각).

### 4.3 UCB 탐색 — "아는 길만 가지 마라"

UCB(Upper Confidence Bound)는 "탐험 vs 활용" 딜레마의 해결책. 레스토랑 선택에 비유하면:

- **활용**: 맨날 가던 맛집에 간다 (안전하지만 새로운 발견 없음)
- **탐험**: 안 가본 곳에 가본다 (불확실하지만 더 좋은 곳을 찾을 수도)

UCB는 이 둘의 균형을 수학적으로 잡아준다.

**mcp-memory에서:** recall할 때, 연결이 강한 기억만 가져오면 항상 같은 결과가 나온다. UCB는 "방문 횟수가 적은 기억"에 보너스를 줘서, 잘 안 쓰이지만 관련성 있는 기억도 발견할 수 있게 한다.

세 가지 모드:
- **focus (c=0.3)**: 검증된 기억 위주 (코딩, 디버깅)
- **auto (c=1.0)**: 균형 (기본 모드)
- **dmn (c=2.5)**: 탐험 극대화 (브레인스토밍, 연결 발견)

### 4.4 기억 재공고화 — "잘 때 정리한다"

뇌는 수면 중에 낮에 경험한 기억을 재구성한다. 새로운 맥락을 붙이고, 관련 기억끼리 통합한다.

**mcp-memory에서:** recall될 때마다, 관련 엣지의 `description` 필드에 "이 맥락에서 함께 사용됨"이라는 기록이 JSON으로 쌓인다. 같은 연결이 다른 맥락에서 사용될수록 그 연결의 "의미 깊이"가 늘어난다.

### 4.5 시냅스 가지치기 — "안 쓰면 잘라낸다"

뇌는 사용하지 않는 시냅스를 제거한다. 10대 때 시냅스 수가 절반으로 줄어드는데, 이건 나빠지는 게 아니라 **효율화**다.

**mcp-memory에서:** 두 종류의 가지치기:
- **엣지 가지치기**: 연결 강도가 0.05 미만이고 30일 이상 안 쓰인 연결 → 아카이브 (삭제가 아님, 복구 가능)
- **노드 가지치기**: 품질 점수 0.3 미만이고 90일 이상 안 쓰이고 연결이 3개 미만인 노드 → 후보 표시 → 30일 유예 → 아카이브

**안전장치**: 최상위 레이어(L4/L5) 기억과 허브 노드(연결이 많은 핵심 노드)는 절대 가지치기 대상이 아니다.

### 4.6 확산 활성화 — "하나를 떠올리면 연결된 것도 따라온다"

"사과"를 떠올리면 "빨간색", "과일", "뉴턴" 같은 연결된 개념이 자동으로 떠오른다. 이게 확산 활성화.

**mcp-memory에서:** recall("포트폴리오 디자인")을 하면, 직접 매칭되는 기억뿐 아니라 그 기억과 연결된 기억, 그리고 그 기억과 또 연결된 기억까지 그래프를 따라 탐색한다 (2-3 hop).

### 4.7 패치 전환 — "같은 밭만 파지 마라"

동물의 먹이 찾기(foraging) 이론에서 가져온 개념. 한 영역(패치)에서 수확이 줄어들면, 새로운 패치로 이동하는 게 전체적으로 더 효율적이다.

**mcp-memory에서:** recall 결과의 75% 이상이 같은 프로젝트에서 나오면 "포화"로 판정. 자동으로 다른 프로젝트의 기억도 섞어서 반환한다. "포트폴리오만 검색했는데 orchestration의 관련 결정도 같이 나옴" — 이게 의도된 동작이다.

## 5. 무엇이 문제인가 — 9개 AI의 진단

현재 mcp-memory v2.0은 기본 구조가 작동하지만, 9개 AI(GPT, Gemini, Perplexity, Grok, DeepSeek, NotebookLM, Elicit x3)에게 심층 분석을 의뢰한 결과 공통으로 지적된 문제가 있다:

### 5.1 학습이 불안정하다

현재의 단순 헤비안 학습은 **연결이 무한히 강해질 수 있다.** 정규화(BCM)가 없어서, 자주 같이 검색되는 기억끼리의 연결이 계속 강해지다가 결국 시스템이 "같은 결과만 반복" 하게 된다.

→ **해결: BCM 규칙 도입.** 레이어별로 다른 학습률(L0=빠르게 적응, L5=거의 변하지 않음). 슬라이딩 임계값으로 과잉 강화 방지.

### 5.2 검증 체계가 없다

기억을 저장할 때 "이 타입이 유효한가?" 를 확인하는 코드가 있지만, **실제로 연결되어 있지 않다.** validators.py가 존재하지만 server.py에서 호출하지 않는다.

→ **해결: validators를 live DB(type_defs 테이블) 기반으로 전환하고, server.py에 직접 연결.** 잘못된 타입은 저장 차단, deprecated 타입은 자동 교정.

### 5.3 허브 보호가 없다

그래프에서 연결이 특히 많은 노드("허브")가 있다. 이 허브가 잘못 수정되거나 삭제되면 그래프 전체가 흔들린다. 현재는 모든 노드에 같은 수준의 보호가 적용된다.

→ **해결: access_control 3계층.** L4/L5 노드 보호 + 상위 10개 허브 노드 쓰기 제한 + 레이어별 권한 테이블.

### 5.4 승격에 근거가 없다

현재 노드를 한 타입에서 다른 타입으로 승격(예: Observation → Pattern)할 때, "타입이 유효한가"만 확인한다. "이 관찰이 정말 패턴인가?"에 대한 **통계적 근거가 전혀 없다.**

→ **해결: 3단계 통계 게이트.** SWR(다양한 맥락에서 검색됐는가?) → Bayesian(충분한 증거가 있는가?) → MDL(의미적으로 유사한 것들과 묶일 수 있는가?). 세 관문을 모두 통과해야 승격.

### 5.5 의미적 피드백 루프 위험

NotebookLM이 발견한 치명적 시나리오: enrichment(LLM이 기억을 풍부하게 만드는 과정)가 **잘못된 정보를 생성** → 그 잘못된 정보가 임베딩에 반영 → 잘못된 연결 생성 → 다음 enrichment에서 그 잘못된 연결을 참조 → **오류가 무한 증폭.**

뇌과학 용어로는 "거짓 기억 공고화(false memory consolidation)"에 해당.

→ **해결: drift 탐지.** enrichment 전후의 임베딩 유사도를 측정하여, 의미가 크게 변하면(cosine similarity < 0.5) 업데이트를 차단.

## 6. 무엇을 구현하는가 — 변화의 전체 그림

### Phase 0: 기반 공사 (1주)

**이전:** 테이블 4개, 검증 미연결, 타입 정의가 코드에 하드코딩.
**이후:** 테이블 10개, 모든 작업을 action_log에 기록, 타입/관계 정의가 DB에 저장, 검증이 실시간 작동.

핵심 변화:
- `action_log` 테이블: 모든 기억 작업(생성, 검색, 승격, 삭제)을 기록. 나중에 "이 기억이 왜 이렇게 됐는지"를 추적 가능.
- `type_defs` / `relation_defs` 테이블: 50개 타입과 48개 관계를 DB에 저장. 코드 수정 없이 런타임에 타입 추가/비활성화 가능.
- `validators.py` 연결: 잘못된 타입의 기억은 저장 시점에 차단.

### Phase 1: 핵심 교체 (2-3주)

**이전:** 단순 헤비안 학습, 기능이 뒤섞인 remember() 함수, 권한 구분 없음.
**이후:** BCM 학습, 분리된 파이프라인, 3계층 접근 제어, 의미 드리프트 탐지.

5개 핵심 파일 전체 교체:
- `hybrid.py`: 헤비안 → BCM+UCB. 연결이 균형 있게 강화/약화되고, 탐색이 다양해짐.
- `remember.py`: 1개 함수 → 4개 함수(classify→store→link). 각 단계를 독립적으로 테스트/재사용 가능. L4/L5 방화벽 내장.
- `recall.py`: mode 파라미터 추가, 패치 포화 전환 도입.
- `action_log.py`: 25가지 작업 유형을 기록하는 신규 모듈.
- `access_control.py`: 레이어/허브 기반 3계층 보호.

### Phase 2: 고급 기능 (4-6주)

**이전:** 즉시 승격(아무 근거 없이), 가지치기 없음.
**이후:** 통계 기반 승격(평균 8.2회 검색 후 판정), 자동 가지치기.

- `promote_node.py`: SWR→Bayesian→MDL 3단계 게이트. "이 Observation이 정말 Pattern인가?"를 통계적으로 증명해야 승격.
- `daily_enrich.py`: Phase 6 가지치기 추가. 매일 실행되는 enrichment에 엣지→노드 순서로 정리.
- `hub_monitor.py`: 허브 노드 건강 상태 주기적 스냅샷.

### Phase 3: 검증과 튜닝

**목적:** "실제로 잘 작동하는가?"를 측정.

- 골드셋 25개 쿼리로 recall 품질(NDCG) 측정
- SPRT 시뮬레이션으로 승격 파라미터 검증
- drift threshold 캘리브레이션
- RRF k=30 vs k=60 A/B 테스트

## 7. 결과: 어떤 시스템이 되는가

모든 구현이 완료되면, mcp-memory는 이렇게 작동한다:

**기억 저장(remember):**
```
사용자: "BCM 직행으로 결정했다"
  → classify: "이건 Decision이다, L1, tier 자동 배정"
  → store: SQLite에 노드 저장, ChromaDB에 임베딩 저장
  → link: 유사한 기억을 찾아 자동 연결 (단, L4/L5 보호됨)
  → action_log: "node_created" 기록
```

**기억 검색(recall):**
```
사용자: "BCM 관련 결정 찾아줘"
  → 벡터 검색 + 텍스트 검색 + 그래프 탐색 (3가지 동시)
  → BCM 학습: 함께 검색된 기억들 사이 연결 강화/약화
  → UCB: 잘 안 쓰인 관련 기억도 추천
  → 패치 전환: 같은 프로젝트만 나오면 다른 프로젝트도 혼합
  → action_log: "recall" + "node_activated" 기록
```

**기억 성장(promotion):**
```
Signal 노드가 8회 이상 검색됨
  → SWR: "벡터/텍스트 둘 다에서 검색됐나? 다양한 프로젝트에서?"
  → Bayesian: "충분한 증거? P(진짜 패턴) > 0.5?"
  → MDL: "비슷한 기억들과 의미적으로 묶이나?"
  → 세 관문 통과 → Pattern으로 승격
  → action_log: "node_promoted" 기록
```

**기억 정리(pruning):**
```
매일 실행:
  → 엣지 가지치기: 약한 연결 식별 → 맥락 다양성 확인 → 아카이브/삭제
  → 노드 가지치기: 저품질+미사용 노드 → 30일 유예 → 아카이브
  → 보호: L4/L5 노드, 허브 노드는 절대 건드리지 않음
  → action_log: 모든 가지치기 기록
```

**전체적으로:** 사용할수록 중요한 기억은 더 잘 연결되고, 불필요한 기억은 자연스럽게 사라지며, 새로운 패턴이 통계적으로 검증되어 올라가는 — **자기조직화하는 기억 시스템.**

## 9. 핵심 용어 사전

| 용어 | 의미 |
|------|------|
| 노드 (Node) | 하나의 기억. 텍스트 + 타입 + 메타데이터. |
| 엣지 (Edge) | 두 기억 사이의 연결. 관계 타입 + 강도(frequency). |
| 그래프 | 노드와 엣지가 이루는 전체 네트워크. |
| 임베딩 (Embedding) | 텍스트를 숫자 벡터로 변환한 것. 의미가 비슷하면 벡터도 가깝다. |
| recall | 기억 검색. 질문을 던지면 관련 기억을 찾아 반환. |
| remember | 기억 저장. 새 노드 생성 + 자동 엣지 생성. |
| enrichment | LLM이 기존 기억에 요약/키워드/패싯을 추가하는 과정. |
| 승격 (Promotion) | 낮은 레이어 → 높은 레이어로 타입 변경. |
| 가지치기 (Pruning) | 사용하지 않는 연결/노드를 정리하는 과정. |
| 허브 (Hub) | 연결이 특히 많은 핵심 노드. |
| drift | enrichment 후 의미가 크게 변한 것. 오류 가능성 신호. |
| frequency | 엣지의 가중치. 함께 검색된 횟수에 기반. |
| theta_m (θ_m) | BCM의 슬라이딩 임계값. 노드마다 다른 학습 기준점. |
| RRF | Reciprocal Rank Fusion. 여러 검색 결과를 하나로 합치는 방법. |
| SPRT | Sequential Probability Ratio Test. 관측을 누적하며 판정하는 통계 기법. |

## 10. 이 시스템의 의미

mcp-memory는 단순한 메모장이 아니다. **사용할수록 똑똑해지는 기억 시스템**이다.

인간의 뇌가 하는 것 — 중요한 것을 강화하고, 불필요한 것을 잊고, 패턴을 발견하고, 새로운 연결을 만드는 것 — 을 소프트웨어로 구현한다. 완벽한 모방이 아니라, **뇌의 원리를 차용한 공학적 시스템**이다.

궁극적으로 이것이 가능하게 하는 것: AI와 인간이 장기적으로 함께 일할 때, **맥락을 잃지 않는 것.** 어제의 결정, 지난달의 실패, 반복되는 패턴 — 이 모든 것이 연결된 채로 보존되고, 필요할 때 정확히 떠오르는 것.
