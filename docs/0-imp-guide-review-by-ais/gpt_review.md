# mcp-memory 온톨로지 설계 점검 (비판적 리뷰)

> 대상: 사용자가 작성한 “mcp-memory 온톨로지 가이드”  
> 관점: 장기 운영/유지보수/품질 향상 관점의 비판적 설계 점검  
> 목적: 지금 설계가 장기적으로 버티는지, 실제 recall/결정 품질을 올리는지 확인

---

## 1) 강점 (그대로 살릴 핵심)

- **기억 = 연결 + 강화/약화 + 승격 + 가지치기**로 정의한 점이 견고합니다. 단순 저장소가 아니라 “동역학”이 있는 시스템으로 설계를 고정해두었습니다.
- **action_log를 why trail(왜 이렇게 됐는지 추적)**로 두려는 방향이 맞습니다. 장기 운영에서 가장 큰 리스크는 “왜 이렇게 됐지?”가 풀리지 않는 상태입니다.
- **enrichment 피드백 루프(거짓 기억 공고화) 위험을 정면으로 다룬 점**이 좋습니다. 이 통제가 없으면 “기억 시스템”이 아니라 “자기 환각 증폭기”가 됩니다.
- **모드(focus/auto/dmn) + 패치 전환**은 실사용 UX로 연결되는 좋은 장치입니다. (단, 아래의 “오염 통제”가 함께 있어야 안정합니다.)

---

## 2) 가장 큰 리스크 5개 (무너질 수 있는 지점)

### R1. 타입 50개 / 관계 48개는 “지식”이 아니라 “운영비”가 될 수 있습니다
- **문제:** 분류/관계 부여가 조금만 흔들려도 그래프가 빠르게 “애매한 연결 덩어리”가 됩니다. 특히 LLM 분류기는 48 관계를 안정적으로 구분하기 어렵고, 시간이 갈수록 라벨 의미가 흐려질 가능성이 큽니다.
- **증상 예시**
  - `supports`가 어느 순간 `related_to`처럼 쓰임
  - 같은 상황이 프로젝트/날짜에 따라 다른 타입/관계로 저장됨
  - recall 결과는 풍부해 보이지만 “결정 품질”이 오르지 않음

### R2. BCM/UCB 동역학은 수단이고, “무엇을 최적화하는지”가 더 중요합니다
- **문제:** 강화/약화는 수단입니다. 시스템은 결국 “좋은 recall”이 무엇인지 목표 정의가 있어야 합니다. 목표가 없으면 “뇌 모사”가 “랜덤워크”로 변질될 수 있습니다.
- **증상 예시**
  - 자주 쓰는 기억이 더 자주 나오며(자기강화), 새로운 문제에서 답이 늦어짐
  - dmn 모드가 “창발”이 아니라 “산만함”으로 체감됨

### R3. drift 탐지에 단일 임계값(cosine < 0.5)을 쓰면 오탐/미탐이 커집니다
- **문제:** 임베딩 유사도는 모델/토크나이즈/요약 스타일에 민감합니다. “의미 변화”와 “표현 변화”를 한 숫자로 자르기 어렵습니다.
- **증상 예시**
  - 안전한 enrichment가 막힘(오탐)
  - 위험한 hallucination이 통과(미탐)
  - 운영자가 임계값을 ‘감’으로 만지기 시작(퇴행)

### R4. 승격(3-gate)이 무거워지면, 운영에서는 “승격이 죽을” 수 있습니다
- **문제:** SWR/Bayesian/MDL은 강력하지만, 데이터 요건/파라미터/실패 케이스 정의가 미흡하면 “승격이 거의 안 일어남” 또는 “엉뚱하게 승격” 중 하나로 갑니다.
- **증상 예시**
  - L0/L1에만 계속 쌓이고 L2/L3가 얇아짐
  - 운영자가 수동으로 승격을 뚫기 시작(규칙 붕괴)

### R5. 패치 전환은 강력하지만, 크로스-프로젝트 오염의 주범이 될 수 있습니다
- **문제:** orchestration에서의 좋은 결정이 portfolio에선 잘못된 맥락 주입이 될 수 있습니다. “관련 있는 척하는 잡음”이 늘면 신뢰가 떨어집니다.
- **증상 예시**
  - 프로젝트 경계가 무너져 “왜 이게 나오지?”가 잦아짐
  - 장기적으로 recall 사용량 감소 → 학습 신호 약화

## 3) 설계적으로 “한 단계만 더” 넣으면 안정해지는 장치들

### (A) 타입/관계 폭발 방지: “코어 온톨로지 + 확장” 2단 구조
- **권장 구조**
  - 코어 타입 12~15개만 강제: `Decision / Failure / Pattern / Principle / Observation / Question / Task / Artifact / Metric / Constraint / Actor / Context` 등
  - 나머지는 **facet(태그/속성)**로 이동: `domain=`, `stage=`, `confidence=`, `stability=`, `sensitivity=` 등
- **효과**
  - 분류/일관성이 올라가고, 관계 라벨의 의미가 유지됩니다.
  - “코드/검증/리콜” 전 파이프라인에서 운영 비용이 낮아집니다.

### (B) 레이어는 단일 축보다 최소 3축이 필요합니다
현재 L0~L5는 “깊이/추상도” 한 축에 가깝습니다. 운영에서 중요한 축은 보통 3개입니다.
- **maturity(성숙도):** 관찰 → 패턴 → 프레임워크
- **confidence(확신도):** 근거/반례/재현성 기반 신뢰
- **stability(변경 저항):** 자주 바뀌는가, 장기 원칙인가

**효과**
- “깊이는 높은데 확신은 낮은 철학” 같은 상태를 표현 가능
- 가지치기/승격/보호 규칙이 훨씬 자연스러워집니다

### (C) enrichment는 “원본 불변 + 파생 가변” 분리가 필수입니다
- **원본(immutable):** 사용자가 말한 텍스트(혹은 1차 기록)
- **파생(variant, versioned):** 요약/키워드/설명/연결 제안 등
- **추가 필수 메타**
  - `provenance`: 모델/프롬프트/시점/버전
  - `trust_level`: `human-asserted / model-derived / mixed`

**효과**
- 거짓 기억 공고화를 기술적으로 차단할 수 있습니다.
- drift/감사/롤백이 쉬워집니다.

### (D) UCB 탐험은 “검색 결과에 섞기”보다 “추천 레인” 분리가 안전합니다
- 메인 recall: **정확도 레인(Exploit)**
- UCB: **발견 레인(Explore)**

**효과**
- 답변 컨텍스트 오염을 줄이면서도 탐험 UX를 유지할 수 있습니다.

## 4) “검증 체계”를 진짜로 끝내는 기준

검증은 validators 연결만으로 끝나지 않고, 최소 3종이 필요합니다.

1) **스키마 검증(형식):** 타입/관계/필수 메타 존재  
2) **동작 검증(행동):** remember→link→recall이 일관된 경로로 호출되는지  
3) **품질 검증(성과):** recall이 실제로 좋아졌는지  

특히 3)이 없으면 시스템은 복잡해지기만 하고 “좋아진 느낌”만 남습니다.

## 5) 최우선 순위 제안 (P0~P2)

### P0 (최우선) — “거짓 기억 증폭” 방지
- 원본/파생 분리 + provenance + trust_level
- drift는 단일 cosine 임계값이 아니라 **합성 점수**로:
  - (a) 텍스트 기반 변화량
  - (b) 임베딩 변화량
  - (c) 위험도 규칙(금지어/사실 주장/정합성 체크 등)

### P1 — “무엇이 좋아졌는지” 측정하는 평가 하네스
- 골드셋 25쿼리는 좋습니다. 여기에 **정답의 형태(산출 규격)**를 고정하는 편이 안정합니다.
  - 예: “결정 노드 3개 + 근거 링크 3개 + 최신순 1개” 같은 반환 규격
- 온라인 운영 지표 1개는 반드시 필요합니다.
  - 예: `Time-to-decision(결정까지 걸린 턴 수)` / `Regret(번복 비율)` / `Re-explain rate(재설명 비율)`

### P2 — 타입/관계 압축(코어+facet)으로 운영 안정화
- 50/48을 유지하더라도 **모델이 반드시 쓰게 하는 코어 subset**을 두면 품질이 크게 상승합니다.
- 나머지는 수동/분석/후처리용으로 분리하는 편이 안전합니다.

---

## 6) 성공/실패 판정 기준 (한 줄)

- **성공:** 시간이 지날수록 “설명 재작성”이 줄고, “결정 재현”이 늘어납니다.  
- **실패:** 시간이 지날수록 “검색은 풍부한데 답은 늦고”, 운영자가 규칙을 예외 처리하기 시작합니다.
